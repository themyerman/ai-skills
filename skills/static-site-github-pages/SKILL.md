# Static Sites on GitHub Pages

## Tom's actual setup

Three patterns in use across repos:

| Repo | Source | Published at |
|------|--------|-------------|
| `niwot-film-fest` | Root (`/`) | `niwotfilmfestival.com` via CNAME |
| `creative-tools` | `docs/` subfolder | GitHub Pages URL (workflow writes `docs/index.html`) |
| `victoria-brief` | Root (`/`) | GitHub Pages URL |

All are plain HTML/CSS/JS — no Jekyll, no build step, no templating engine.

---

## Two deployment patterns

### Pattern A: Root deployment (niwot-film-fest, victoria-brief)
The repo root is the site. `index.html` at the root is the homepage. GitHub Pages serves everything directly.

```
repo/
├── index.html
├── about.html
├── css/
│   └── style.css
├── js/
│   └── nav.js
├── img/
└── CNAME          ← custom domain if needed
```

**Pages setting**: Settings → Pages → Source → Deploy from branch → `main` → `/` (root)

### Pattern B: docs/ subfolder (creative-tools)
The repo has other code at root. The site lives in `docs/`. A GitHub Actions workflow generates or copies files there and commits them.

```
repo/
├── src/           ← tool code
├── docs/
│   └── index.html ← generated by workflow
└── .github/
    └── workflows/
        └── publish.yml
```

**Pages setting**: Settings → Pages → Source → Deploy from branch → `main` → `/docs`

## Custom domains (CNAME)

Add a `CNAME` file at the root of your site (root for Pattern A, inside `docs/` for Pattern B) containing just the domain — no protocol, no trailing slash:

```
niwotfilmfestival.com
```

**DNS setup** (at your registrar):
- For an apex domain (`example.com`): four A records pointing to GitHub's IPs:
  ```
  185.199.108.153
  185.199.109.153
  185.199.110.153
  185.199.111.153
  ```
- For a `www` subdomain: CNAME record pointing to `yourusername.github.io`
- For a subdomain (`sub.example.com`): CNAME record pointing to `yourusername.github.io`

DNS changes take minutes to hours to propagate. GitHub enforces HTTPS automatically once DNS resolves — may take up to 24 hours for the cert to provision.

**Pitfall**: if you set the custom domain in GitHub's UI but don't have the CNAME file in the repo, the setting resets on next push. Always keep the CNAME file in the repo.

## GitHub Actions: automated publishing

Use when something generates the site content (a Python tool, a script) and needs to commit the output.

```yaml
name: Publish Site

on:
  schedule:
    - cron: "0 13 * * *"   # daily at 6am MDT
  workflow_dispatch:        # manual trigger

permissions:
  contents: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4.2.2

      - uses: actions/setup-python@v5.6.0
        with:
          python-version: "3.12"
          cache: "pip"

      - run: pip install .

      - name: Generate site
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: your-tool --output docs/index.html

      - name: Commit and push
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
          git add docs/index.html
          git diff --staged --quiet || git commit -m "publish: $(date -u '+%Y-%m-%d')"
          git push
```

**Key details:**
- `permissions: contents: write` is required for the workflow to push
- `git diff --staged --quiet ||` skips the commit if nothing changed (avoids empty commits)
- Use `actions/checkout@v4.2.2` and `actions/setup-python@v5.6.0` — avoid older versions that trigger Node.js deprecation warnings in the Actions runner
- `workflow_dispatch` lets you trigger manually from the GitHub Actions UI without waiting for the schedule

## Disabling Jekyll

GitHub Pages runs Jekyll by default, which can mangle files that start with `_` or directories named `_site`. Add an empty `.nojekyll` file to the root (or `docs/`) to disable it:

```bash
touch .nojekyll
git add .nojekyll
git commit -m "disable Jekyll"
```

Plain HTML sites don't need Jekyll. Always add `.nojekyll` for plain HTML repos.

## HTML patterns for multi-page sites

Without a build system, keep navigation consistent by hand. A simple approach:

```html
<nav>
  <div class="wrap">
    <a class="nav-brand" href="index.html">Site Name</a>
    <ul class="nav-links">
      <li><a href="about.html">About</a></li>
      <li><a href="faq.html">FAQ</a></li>
    </ul>
  </div>
</nav>
```

- Use relative paths (`href="about.html"`, not `/about.html`) so the site works both on GitHub Pages and opened locally from the filesystem
- Mark the current page active: `<a href="about.html" class="active">About</a>`
- Keep a consistent footer with copyright, links, and any required attributions

**Images and assets**: put them in `img/`, reference them relatively. GitHub Pages serves them as static files.

## Google Analytics

Standard gtag snippet goes in `<head>` of every page:

```html
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXX');
</script>
```

Replace `G-XXXXXXXX` with the measurement ID from Google Analytics → Admin → Data Streams.

## Open Graph / social sharing

Add to `<head>` of every page so links preview correctly in social and messaging apps:

```html
<meta property="og:title" content="Page Title">
<meta property="og:description" content="One sentence description.">
<meta property="og:image" content="https://yourdomain.com/img/og-image.jpg">
<meta property="og:url" content="https://yourdomain.com/page.html">
<meta property="og:type" content="website">
<meta name="twitter:card" content="summary_large_image">
```

OG image should be at least 1200×630px. Use an absolute URL (not relative) — social crawlers won't follow relative paths.

## Troubleshooting

**Site not updating after push**
- GitHub Pages can take 1–2 minutes to deploy. Check Actions tab for build status.
- Hard-refresh the browser (Cmd+Shift+R / Ctrl+Shift+R) to bypass local cache.
- Check Settings → Pages to confirm the source branch and folder are correct.

**404 on all pages except index**
- Confirm Pages is set to the right branch and folder.
- Check for a Jekyll conflict — add `.nojekyll`.
- Check that filenames match exactly (GitHub Pages is case-sensitive on Linux runners).

**Custom domain not working**
- Verify the CNAME file is in the repo (not just set in the GitHub UI).
- Check DNS propagation with `dig yourcustomdomain.com` or `nslookup`.
- HTTPS cert provisioning can take up to 24 hours after DNS resolves.

**Workflow fails to push**
- Confirm `permissions: contents: write` is in the workflow YAML.
- If pushing to a protected branch, you may need to allow GitHub Actions to bypass branch protection in Settings → Branches.

**Site shows raw Jekyll output or `---` at top**
- Jekyll is processing your files. Add `.nojekyll` to the root.

## Quick reference: Pages settings location

GitHub repo → **Settings** → **Pages** (left sidebar under "Code and automation")

- **Source**: Deploy from branch
- **Branch**: `main` (or `gh-pages`)
- **Folder**: `/` (root) or `/docs`
- **Custom domain**: enter domain here AND keep CNAME file in repo
